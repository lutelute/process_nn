## 1. 初回レビュー時点の総合判断

このプロジェクトの強みは、74 個の可視化ページと GPT-2 ビューアを、共通の学習導線と落ち着いたデザインでまとめた点にある。デスクトップとモバイルの入口、5 ステップの進行、誤解をほどく欄、前後テーマへの導線はよくできており、「まず触って全体像をつかむ教材」としてかなり魅力がある。

一方、技術教材としての最大の課題は、説明が実装より強くなっているページがあることだ。特に拡散モデルと RAG は、現在の実装のままでは中核となる説明と計算が一致していない。BatchNorm、オートエンコーダ、VAE、線形回帰、MoE にも、誤解を生みやすい断定がある。GPT-2 は構成自体は意欲的だが、本物の重みを用いた再現可能な検証がリポジトリに残っておらず、最大文脈でのメモリ・描画設計にも実用上の問題がある。

したがって、次の優先順位を推奨する。

1. 拡散モデルと RAG の「実装と説明の不一致」を直す
2. BatchNorm、AE/VAE、線形回帰、MoE の断定を正確にする
3. 数値主張にテスト・シード・結果ログ・出典を付ける
4. 基礎数学、評価、データ実務の教材を補う
5. その後に新しい応用手法を増やす

## 3. 優先度 P0 — 内容の中核と実装が一致しない

### P0-1. 拡散モデルの終端が純ノイズにならない

対象: `viz/diffusion.html:90,132-155,206-208`

実装は、ステップ数 `T = 20 / 40 / 80` に対して、元の DDPM と同じ `beta = 1e-4 ... 0.02` の線形スケジュールをそのまま使っている。終端の累積信号量 `alpha_bar_T` は次の通りで、0 から大きく離れている。

| T | `alpha_bar_T` | 元信号の係数 `sqrt(alpha_bar_T)` | ノイズ係数 |
|---:|---:|---:|---:|
| 20 | 0.8168 | 0.9038 | 0.4280 |
| 40 | 0.6671 | 0.8168 | 0.5769 |
| 80 | 0.4451 | 0.6672 | 0.7449 |

たとえば既定の T=40 では、終端にも元データが係数 0.82 で残る。したがって、教材内の「最後はただのガウスノイズ `N(0,I)`」という説明は、この実装には当てはまらない。さらに学習はこの信号が残った `x_t` で行う一方、生成は `x_T ~ N(0,I)` から開始しているため、学習時と生成時の入力分布も一致しない。

元の DDPM は同じ beta 端点を T=1000 で使用し、終端の事前分布との KL がほぼ 0 になるようにしている。式だけを借りて T を 20–80 に短縮する場合、終端の `alpha_bar` が十分小さくなるようスケジュールを再設計する必要がある。参考: [Denoising Diffusion Probabilistic Models](https://arxiv.org/pdf/2006.11239)。

推奨判断: 最優先。短いステップ用のスケジュールへ変更して再学習・結果検証するか、「完全な DDPM」ではなく信号の残る簡略デモであることを明記する。

### P0-2. RAG の「回答生成」は生成していない

対象: `viz/rag.html:132-189,224-238`

文字バイグラム TF-IDF、コサイン類似度、上位 k 件の検索は実際に計算されている。一方、回答は質問データに最初から保存された `Q.a` をそのまま表示しており、取得チャンクを入力にした生成処理はない。表示上の出典だけが、その時点の検索上位文書から作られる。

このため、検索が誤っても正解の定型回答を表示できる。「取得文に基づいて答える」「文書に書かれていないことは答えない」という説明は、現実の RAG 一般にも現在の実装にも保証されない。元論文での RAG は retriever と、取得文書に条件づけられた seq2seq generator を組み合わせる。参考: [Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks](https://arxiv.org/abs/2005.11401)。

推奨判断: 最優先。現状を「検索＋事前作成回答の表示」と正直にラベル付けするか、回答抽出・生成を取得チャンクから実際に行う。検索を外した場合、根拠不足の場合、文書にない質問の場合も教材に入れると教育価値が高い。

## 4. 優先度 P1 — 誤解を生みやすい説明・未検証の主張

### P1-1. BatchNorm ページは「学習が速い」ことを実演していない

対象: `viz/batchnorm.html:63-68,104-126,148-158`、`README.md:42`

ページ内で行っているのは、固定したランダム重みの順伝播に BN を挟み、層ごとの活性の標準偏差を比較することだけである。loss、逆伝播、重み更新、収束速度の比較はなく、`gamma=1.4`、`beta=-0.6` も学習値ではない。したがって、「速く安定して学習」を結果として示したページではなく、「活性スケールをそろえる計算」のデモである。

また、内部共変量シフトを主因として断定しているが、これは原論文の動機ではあっても、効果の説明として決着したものではない。後続研究は、分布安定化との因果を否定する実験と、最適化地形・勾配を滑らかにする説明を提示している。参考: [How Does Batch Normalization Help Optimization?](https://proceedings.neurips.cc/paper/2018/hash/905056c1ac1dad141560467e0a99e1cf-Abstract.html)。

推奨判断: 「BN の計算と活性スケール」のページとして説明を弱めるか、同じ初期値・複数 seed で BN 有無の訓練曲線を実測する。`gamma/beta` と推論時 running statistics も実装すると説明と一致する。

### P1-2. リングを通常の 1 次元潜在へ無損失圧縮できる、は一般には誤り

対象: `viz/autoencoder.html:106,119-120,158-163`

S 字のような区間状の 1 次元多様体と、閉じたリング `S^1` は位相が異なる。連続な encoder / decoder と通常の実数 1 次元潜在 `R` では、リング全体を継ぎ目なしに一対一で表すことはできない。どこかに切れ目、重なり、または再構成誤差が必要になる。「本質次元が 1 なら 1 次元 z でほぼ無損失」という説明は、位相条件を落としている。参考: [Topological Obstructions to Autoencoding](https://arxiv.org/abs/2102.08380)。

推奨判断: リングをむしろ反例として使い、S 字との違い、潜在の継ぎ目、円周潜在 `(sin theta, cos theta)` または 2 次元潜在を見せる。

### P1-3. 線形回帰の「厳密解」「一意」の条件が抜けている

対象: `README.md:37`、`viz/linreg.html:98-106`

- 勾配降下と正規方程式は同じものではない。勾配降下は、適切な学習率で十分反復したときに同じ最小解へ近づく方法。
- 二乗損失が凸であるだけでは解は一意にならない。一意性には、典型的には設計行列の full column rank、すなわち厳密凸性が必要。
- `(X^T X)^-1 X^T y` は `X^T X` が特異なら使えず、一般説明では擬似逆行列または正則化が必要。

このページの単回帰データ自体は通常 full rank なので、実演結果ではなく一般化した説明の問題である。

### P1-4. VAE の KL は「穴なし」「すべての z が意味を持つ」を保証しない

対象: `viz/vae.html:163-167`

KL 項は各 `q(z|x)` を事前分布へ近づけるが、集約事後分布と事前分布の完全一致、穴の不存在、任意の z の良好な復号を保証しない。latent holes と posterior collapse は VAE の既知の問題である。参考: [Matching Aggregate Posteriors in the Variational Autoencoder](https://arxiv.org/abs/2311.07693)。

推奨判断: 「穴を減らし、サンプリングしやすくするよう促す」に表現を変更し、再構成とのトレードオフと失敗例を見せる。

### P1-5. GPT-4 を MoE の採用例として扱う根拠がない

対象: `README.md:78`、`viz/moe.html:67-69`

Mixtral が MoE であることは公式資料で確認できる。一方、OpenAI の GPT-4 Technical Report は、モデルサイズを含む追加のアーキテクチャ詳細を公開していない。したがって GPT-4 の MoE 採用は、教材の事実欄に置ける確認済み情報ではない。参考: [GPT-4 Technical Report](https://cdn.openai.com/papers/gpt-4.pdf)、[Mixtral 8x7B model card](https://docs.mistral.ai/models/model-cards/mixtral-8x7b-0-1)。

推奨判断: GPT-4 は削除するか、「公開情報では未確認の推測」と明確に分離する。

### P1-6. GPT-2 forward の主張は、現在のリポジトリだけでは再現確認できない

対象: `gpt2/index.html:221-223,332-360,404-440`、`gpt2/tools/gpt2.mjs:113-218`、`gpt2/GPT_NOTES.md:118-124`

確認できた tokenizer テストはよい。一方、重み本体はリポジトリに含まれず、forward の既知 logits、PyTorch 参照値との誤差、生成 smoke test は自動テストに残っていない。そのため「コンポーネント単体テストと本物重みでの生成健全性で確認」という記述は、第三者が再現できない。

さらに、実装から次のリスクを確認した。

- 生成 1 トークンごとに全プロンプトを最初から forward し直し、KV cache がない。
- `capture:true` では全 12 層・12 ヘッドの `T x T` attention を保持する。T=1024 では attention だけで約 604 MB（decimal、576 MiB）。約 548 MB の F32 重みや中間活性は別に必要。
- 各層で logit lens の全語彙射影を行い、さらに最終射影を行うため、12 回分の追加 `lm_head` 計算がある。
- 単一 attention Canvas は `floor(360/T)` をセル幅にするため、T>360 で 0 となり表示できない。
- 「1 トークン数百 ms〜数秒」は、この素の JS、全内部 capture、長い文脈という条件では根拠が保存されていない。
- `GPT_NOTES.md` は「Phase 0 完了、次は forward 実装」のままで、現在の実装状態と矛盾する。

推奨判断: 正しさの判定と性能の判定を分ける。小さな固定 prompt に対する層別 checksum / logits の参照テスト、実測環境・時間・メモリ、対応可能な推奨文脈長を記録する。最大 1024 を UI 上の実用上限として見せない方が安全。

## 5. 優先度 P2 — 再現性、アクセシビリティ、保守性

### P2-1. 数値主張の証拠が教材内・リポジトリ内に残っていない

README には転移学習、蒸留、GNN、PINN、Actor-Critic など多くの具体的な改善率があるが、それが「この toy 設定・この seed の結果」なのか、一般的なベンチマークなのかを一見して区別しにくい。ページ内の単一実行結果を一般的な効果量に見せないことが重要である。

各ページに最低限、次を持たせるとよい。

- Scope: このデモが再現するもの / 省略するもの
- Reproducibility: seed、データ数、反復数、評価式
- Evidence: 期待範囲を検査する自動テスト、複数 seed の中央値とばらつき
- Sources: 原論文または公式資料へのリンク
- Limitations: この結果を一般化できない条件

### P2-2. reduced-motion 対応が重い計算を同期実行してしまう

`dronerace.html:649,657`、`dronerace2.html:573,579,584`、`gnnflow.html:475,482` では、アニメーションを省略する代わりに、学習やレースを `while (...) {}` / `trainAll()` で一度に完了させる分岐がある。動きを抑える目的には合うが、メインスレッドを長時間占有し、操作不能になる。reduced motion と reduced computation は別の設計問題であり、チャンク実行してイベントループへ制御を返す必要がある。

### P2-3. Canvas 教材の代替情報が不足

`viz/nav.js` は Canvas に汎用 `role="img"` とラベルを付けており、最低限の改善にはなっている。しかし、グラフの現在値、結論、学習結果は読み上げられない。トップと GPT-2 のフォームには `for` のない label も多い。

推奨判断: Canvas と同じ情報を短いテキスト要約・表でも更新し、その領域へ適切に `aria-live` を使う。フォームは label と control を ID で関連付け、キーボードだけで全操作を確認する。

### P2-4. CI と文書に件数・進捗のドリフトがある

- `.github/workflows/verify.yml:17` は「全 50 ページ」だが実際は 76。
- `viz/nav.js:5,33` は「全 72 ページ」。
- `gpt2/GPT_NOTES.md:118-124` の進捗は forward 実装前のまま。
- `tools/check-pages.mjs` のリンク検査は二重引用符の `href/src` に限定され、DOM、外部 URL、ユーザー操作、数値結果は見ない。
- Playwright の依存バージョンはコメントにだけあり、manifest / lockfile がない。

件数を手書きで複数箇所へ置かず、検証結果から生成するか「全ページ」とだけ書く方が保守しやすい。

### P2-5. その他の軽微な内容・実装上の注意

- `viz/gmm.html` は共分散へ `2e-3` の正則化を加えながら「EM なので対数尤度が単調増加」と断定する。正則化込みの更新が同じ目的関数の厳密な EM かを区別した方がよい。
- `viz/dqn.html` の reset は一度別 seed を計算した後に `seed=7` へ戻しており、毎回同じ初期化になる。再現性目的なら明記し、別試行目的なら意図と不一致。
- README の「全ページがランダム重みでなく実学習」は、構造説明や固定順伝播を行うページまで含めると強すぎる。「学習が主題のページでは実計算・実学習」とする方が正確。
- 48 個の可視化ページには `class="read"` の数式読み解き行がなく、新旧フォーマットの深さに差がある。全ページを同じ量へ揃える必要はないが、必須項目を定義すると品質を保ちやすい。

## 6. 追加すると価値が高い素材

現状は応用手法の幅が非常に広い。次はアルゴリズムの種類を増やすより、既存 74 ページをつなぐ「橋」を足す方が、初学者向けロードマップとして効果が大きい。

### 最優先: 数学と計算の前提

1. ベクトル、行列、shape、内積、行列積
2. 微分、偏微分、連鎖律、計算グラフ
3. 確率分布、期待値、最尤法、対数尤度
4. entropy、cross-entropy、KL divergence の関係

数式を増やす目的ではなく、既存ページの値を同じ表記で追えるようにする「道具箱」として作るのがよい。

### 最優先: モデルを正しく測る素材

1. 回帰指標: MAE / RMSE / R-squared と外れ値・単位の違い
2. 確率校正: reliability diagram、Brier score、しきい値選択
3. ROC と PR の使い分け、クラス不均衡
4. 複数 seed、誤差棒、信頼区間、「差がある」と言える条件
5. ベースライン、ablation、train / validation / test の役割

`metrics.html` と `crossval.html` の続編として置けば、既存の具体的な改善率を批判的に読めるようになる。

### 高優先: 実務の学習・データ工程

1. 標準化、欠損値、カテゴリ変数、前処理パイプラインと leakage
2. 初期化、学習率 schedule、early stopping、gradient clipping
3. 学習曲線から underfit / overfit / optimization failure を切り分ける方法
4. データドリフト、OOD、監視、再学習の判断
5. fairness / bias と「相関と因果」の区別

### 高優先: LLM ページをつなぐ素材

1. tokenizer → 次トークン事前学習 → SFT → preference optimization の全体図
2. decoding: greedy / temperature / top-k / top-p と品質・再現性
3. KV cache、文脈長、計算量・メモリ
4. retrieval、reranking、generation、citation verification の分離
5. RAG が幻覚をなくすのではなく、根拠を与えて検証可能性を高める仕組みであること

### 横断素材: 出典と簡略化のレイヤー

新規ページとは別に、各教材の末尾へ共通の小さな「このデモの前提」欄を加えるのが最も費用対効果が高い。

- 原論文 / 公式資料
- 実際の手法から省略した部分
- toy data と実データの違い
- そのページの数値を一般化してはいけない理由
- seed と再現手順
